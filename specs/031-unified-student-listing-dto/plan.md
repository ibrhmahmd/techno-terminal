# Implementation Plan: Unified Student Listing DTO

**Branch**: `031-unified-student-listing-dto` | **Date**: 2026-06-10 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/031-unified-student-listing-dto/spec.md`

---

## Summary

Standardize all five `GET /crm/students/*` listing endpoints to return a consistent `StudentListingDTO` shape. The unified DTO adds `age` (computed), `date_of_birth`, `gender`, `current_group_name`, and `has_unpaid_balance` to every endpoint that was previously missing them. The filter endpoint's `has_unpaid_balance` query parameter is renamed to `has_any_outstanding_balance` for semantic clarity. This is a pure additive change to the API surface (no field removals) plus two coordinated renames that require simultaneous frontend migration.

---

## Technical Context

**Language/Version**: Python 3.10+  
**Primary Dependencies**: FastAPI, SQLModel, SQLAlchemy (raw SQL via `text()`), Pydantic v2  
**Storage**: PostgreSQL — existing views `v_unpaid_enrollments` and `v_enrollment_balance` used for balance queries  
**Testing**: pytest, existing `tests/` suite, `override_auth` fixture for auth bypass  
**Target Platform**: Leapcell (Linux server)  
**Project Type**: Web service (FastAPI REST API)  
**Performance Goals**: `GET /crm/students?limit=50` must not regress by more than 20% vs. pre-change baseline  
**Constraints**: No new DB migrations required. No schema changes. EXISTS subquery approach keeps query cost bounded.  
**Scale/Scope**: ~200–500 students typical; pool_size=10, max_overflow=5 (15 total connections)

## Phase 0: Research — COMPLETE

See [`research.md`](./research.md) for full findings. Key decisions:

- `has_unpaid_balance` on directory/search/grouped/filter responses: EXISTS subquery on `v_unpaid_enrollments`
- `has_unpaid_balance` on waiting-list: EXISTS subquery on `v_enrollment_balance` (all enrollment history)
- `age`: computed post-query via `StudentValidator.compute_age()`
- Filter parameter renamed: `has_unpaid_balance` → `has_any_outstanding_balance` (semantics preserved: all-enrollment scope)
- No DB migrations needed — all referenced views already exist
